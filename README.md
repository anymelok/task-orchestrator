# Orchestrator Backend Service

Отказоустойчивый бэкенд-сервис оркестрации задач на парке хостов (≥ 1000) с нестабильной сетью, поддержкой ручного согласования, блокировок хостов и встроенной системой мониторинга (Prometheus + Grafana)

---

### 1. Быстрый старт

#### Требования:
*   Установленные `Docker` и `Docker Compose` [1].
*   Утилита `uv` (рекомендуется для локального запуска тестов и утилит разработки) [2].

#### Шаг 1. Клонирование и запуск инфраструктуры
```bash
git clone <url_вашего_репозитория>
cd task-orchestrator

# Запуск всего стека из 8 контейнеров (Django, Celery, MySQL, Redis, Exporter, Prometheus, Grafana)
docker compose up -d --build
```
*Миграции базы данных MySQL и подготовка дашборда Grafana выполняются автоматически при старте* [1].

#### Шаг 2. Запуск нагрузочных тестов (Headless Locust)
Нагрузочное тестирование запускается одной командой прямо с хост-компьютера [4]:
```bash
# С дефолтными параметрами (150 пользователей, 10 новых пользователей в секунду):
make load-test

# С кастомными параметрами (например, 300 пользователей, 20 новых пользователей в секунду):
make load-test USERS=300 RATE=20
```

#### Шаг 3. Вход в панели мониторинга
*   **Grafana** (Визуализация): `http://localhost:3000` (Логин: `admin`, Пароль: `admin`). Дашборд с графиками RPS, длиной очередей и статусами воркеров уже импортирован [1].
*   **Prometheus** (Сбор метрик): `http://localhost:9090` [1].

### 2. Примеры запросов и использование `make`

#### Использование `make` для разработки:
В корне проекта находится `Makefile` для автоматизации проверок стиля и запуска тестов [4]:
*   `make lint` — запуск линтера Ruff.
*   `make lint-fix` — автоисправление простых предупреждений линтера Ruff.
*   `make format` — форматирование кода по стандарту PEP 8 через Ruff.
*   `make fix` — запуск форматирования и исправления ошибок линтера за один шаг.
*   `make typecheck` — проверка статической типизации через Mypy.
*   `make test` — запуск 21 интеграционного теста в изолированном окружении с замером покрытия.
*   `make check` — запуск всей цепочки проверок качества кода (линтер, типы, тесты).

#### Примеры API-запросов (curl):

*   **Отправка задачи на выполнение (вебхук с явным указанием хостов):**
    ```bash
    curl -X POST http://127.0.0.1:8000/webhook/jobs \
      -H "Content-Type: application/json" \
      -d '{
        "external_id": "job-ext-uuid-1",
        "command_type": "PING",
        "payload": {"count": 4},
        "hosts": ["host-db-01", "host-web-02"]
      }'
    ```

*   **Отправка задачи по селектору (тегам в метаданных):**
    ```bash
    curl -X POST http://127.0.0.1:8000/webhook/jobs \
      -H "Content-Type: application/json" \
      -d '{
        "external_id": "job-ext-uuid-2",
        "command_type": "RUN_SCRIPT",
        "payload": {"script": "uptime"},
        "selector": {"dc": "ams", "role": "db"}
      }'
    ```

*   **Получение агрегированного статуса задачи:**
    ```bash
    curl http://127.0.0.1:8000/jobs/1
    ```

*   **Детализация выполнения по хостам (с пагинацией и фильтром по статусу):**
    ```bash
    curl "http://127.0.0.1:8000/jobs/1/executions?status=SUCCESS&page=1"
    ```

*   **Просмотр логов конкретного запуска:**
    ```bash
    curl http://127.0.0.1:8000/executions/1/logs
    ```

*   **Одобрение зависшей задачи (Approval):**
    ```bash
    curl -X POST http://127.0.0.1:8000/jobs/1/approve
    ```

*   **Отклонение зависшей задачи (Reject):**
    ```bash
    curl -X POST http://127.0.0.1:8000/jobs/1/reject
    ```

*   **Отмена задачи в процессе выполнения (Cancel):**
    ```bash
    curl -X POST http://127.0.0.1:8000/jobs/1/cancel
    ```

*   **Установка черного списка команд на хосте:**
    ```bash
    curl -X PUT http://127.0.0.1:8000/hosts/1/blocks \
      -H "Content-Type: application/json" \
      -d '{"command_types": ["RESTART_SERVICE", "DEPLOY"]}'
    ```

*   **Удаление команды из черного списка хоста:**
    ```bash
    curl -X DELETE http://127.0.0.1:8000/hosts/1/blocks/DEPLOY
    ```

### 3. Описание архитектуры и ключевых решений

Архитектурный каркас системы построен на базе **Django ORM** + **MySQL** + **Celery** + **Redis** [1].

#### Ключевые решения:
*   **Идемпотентность**: Реализована на уровне транзакций базы данных с помощью уникального индекса на поле `external_id` в таблице `Job`. В случае конкурентного race condition база данных выбрасывает `IntegrityError`, который перехватывается на уровне представления, исключая дублирование.
*   **Блокировки и очереди хостов (Concurrency Control)**: 
    Для исключения параллельного выполнения задач на одном хосте используется распределенная блокировка в Redis (`lock:host:<host_id>`). 
    Чтобы избежать холостой прокрутки воркеров при блокировке (Busy Waiting), задачи выстраиваются в списки ожидания внутри Redis (`host:queue:<host_id>`). Когда текущая задача освобождает лок, она автоматически запускает из очереди следующую задачу, убирая лишнюю нагрузку с воркеров.
*   **Исключение Row Locks на MySQL**:
    При обработке задач на тысячах хостов обращение воркеров к строке `Job` для обновления общего статуса приводило бы к взаимным блокировкам в MySQL. Для решения этой проблемы используется атомарный декрементный счетчик в Redis (`job:remaining:<job_id>`) [3]. Только воркер, опустивший счетчик до `0` (последний завершившийся хост), инициирует единственный, финальный запрос обновления статуса `Job`.
*   **Ретраи по типам ошибок (Network vs Permanent)**:
    Реализовано разделение ошибок на временные и постоянные. При возникновении сетевых сбоев (`AgentConnectionError`) Celery перезапускает задачу с экспоненциальной задержкой. При критических ошибках на стороне агента (`AgentExecutionError`) задача переводится в статус `FAILED` без холостых повторов.
*   **Таймауты выполнения**:
    Реализовано связывание таймаутов на исполнение с лимитами времени Celery (`time_limit` и `soft_time_limit`). При превышении лимита Celery генерирует исключение `SoftTimeLimitExceeded`, что позволяет воркеру корректно освободить Redis-лок хоста и записать информацию о таймауте в базу данных.
*   **Сквозное логирование (Correlation ID)**:
    Используется связка стандартного `logging.Filter` и `contextvars` [1]. Контекст `job_id` автоматически привязывается к асинхронным потокам воркеров и выводится во всех логах (включая сторонние библиотеки) без необходимости ручной передачи аргументов логгеру [1].

#### Соответствие архитектурным ожиданиям:
*   *Стек*: Полностью соответствует рекомендациям (Django, MySQL, Celery, Redis) [1].
*   *Использование Gunicorn*: Вместо отладочного `runserver` в докере запущен промышленный веб-сервер Gunicorn с пулом воркеров и потоков (`--workers 4 --threads 2`), что решает проблему блокировок I/O при высокой конкурентности входящих запросов.

### 4. Адреса подключенных интерфейсов

При поднятом Docker Compose все контейнеры работают во внутренней сети, пробрасывая порты наружу [1]:
*   **Django-бэкенд (API)**: `http://localhost:8000`
*   **Интерфейс Prometheus**: `http://localhost:9090`
*   **Интерфейс Grafana**: `http://localhost:3000` (логин/пароль: `admin` / `admin`, дашборд уже предустановлен)
*   **Celery Exporter**: `http://localhost:9808/metrics` (сбор метрик очередей из Redis) [1.2.7]

#### Запуск Locust с веб-интерфейсом:
Если вы не хотите использовать автоматический headless-тест через `Makefile` [1]:
1.  Запустите Locust в обычном режиме на хосте:
    ```bash
    uv run locust -f locustfile.py
    ```
2.  Откройте браузер: `http://localhost:8089` [1].
3.  Задайте параметры нагрузки и укажите хост `http://127.0.0.1:8000` [1].

### 5. Функциональные требования и документация API

#### Прием задач по вебхуку:
*   `POST /webhook/jobs` — принимает данные задачи, валидирует типы, разрешает хосты по списку или селектору метаданных. 
    *   *Ответ*: `201 Created` с ID задачи при новом запуске; `200 OK` с возвратом старого ID при повторном дубликате.

#### Управление жизненным циклом (Согласование):
*   `POST /jobs/{job_id}/approve` — ручное подтверждение задачи, переведенной в состояние `WAIT_APPROVAL`. Переводит задачу в очередь.
    *   *Ответ*: `200 OK`.
*   `POST /jobs/{job_id}/reject` — отклонение задачи. Переводит её в `CANCELLED`.
    *   *Ответ*: `200 OK`.

#### Отмена задач:
*   `POST /jobs/{job_id}/cancel` — каскадная отмена выполнения. Переводит незапущенные executions в `CANCELLED`.
    *   *Ответ*: `200 OK`.

#### Блокировки хостов:
*   `PUT /hosts/{host_id}/blocks` — запись списка заблокированных типов команд для хоста.
    *   *Ответ*: `200 OK`.
*   `DELETE /hosts/{host_id}/blocks/{command_type}` — снятие блокировки конкретной команды.
    *   *Ответ*: `200 OK`.

#### Мониторинг выполнения:
*   `GET /jobs/{job_id}` — получение агрегированной статистики задачи (сколько успехов, ошибок, в процессе).
    *   *Ответ*: `200 OK`.
*   `GET /jobs/{job_id}/executions` — список хостов-выполнений. Поддерживает пагинацию (`?page=...`), фильтрацию по статусу (`?status=...`) и сортировку (`?sort_by=...`). Оптимизирован против N+1 и не выкачивает тяжелые логи списком [2].
    *   *Ответ*: `200 OK`.
*   `GET /executions/{execution_id}/logs` — получение подробного вывода/логов работы конкретного хоста.
    *   *Ответ*: `200 OK`.

### 6. Реализованные усложнения и симуляция агента

#### Реализованный объем усложнений (Бонусы):
1.  **Селектор хостов по тегам**: возможность поиска хостов по совпадению JSON-метаданных (`metadata__{key}=value`).
2.  **Отмена Job API**: полноценный эндпоинт отмены задач с остановкой очередей и executions.
3.  **Логирование аудита (Audit Log)**: запись всех критических изменений состояний (аппрувы, блокировки, отмены) в отдельную лог-таблицу.
4.  **Разделение типов ошибок**: дифференциация сетевых (ретрируемых) сбоев и критических ошибок выполнения скрипта.

#### Симуляция агента (`MockAgentClient`):
Клиент симулирует реальное нестабильное поведение хостов и сети.
*   **В режиме тестирования (`TESTING = True`)**: агент работает абсолютно детерминировано (всегда возвращает мгновенный успех без задержек `time.sleep`), что обеспечивает прохождение 21 теста за 0.36 секунды без "миганий" [5].
*   **В рабочем режиме (Docker/Prod)**: агент имеет встроенную 25% вероятность случайного отказа (10% — обрыв связи, 5% — таймаут, 10% — код возврата отличный от нуля).

### 7. Структура проекта и запуск миграций

#### Где что находится:
*   `orchestrator/models.py` — ORM-модели сущностей (`Host`, `Job`, `Execution`, `Approval`, `AuditLog`, `HostBlock`) и объявления перечислений статусов.
*   `orchestrator/tasks.py` — логика распределенных блокировок, очередей Redis, планирования и периодического Санитара очистки зависших задач.
*   `orchestrator/agent.py` — абстрактный интерфейс взаимодействия с хостами и его симуляция.
*   `orchestrator/views.py` — эндпоинты API, транзакционная идемпотентность и логирование аудита.
*   `orchestrator/tests/` — разделенная по модулям папка с 21 интеграционным тестом.
*   `pyproject.toml` / `uv.lock` — конфигурация зависимостей, Ruff, Mypy и Pytest.
*   `Makefile` — автоматизация команд линтинга, типизации, тестирования и запуска нагрузки [4].

#### Запуск миграций:
Накат миграций базы данных MySQL выполняется **автоматически** при старте контейнера `web` перед запуском веб-сервера Gunicorn [1]. Порядок строго контролируется Docker Healthcheck — миграции запустятся только тогда, когда MySQL перейдет в стабильное состояние `healthy` [1]. При локальной разработке миграции можно выполнить вручную через `uv run python manage.py migrate`.
